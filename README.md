# 本当に最小のAI - tiny.py 解説

## 全体の流れ

```
文字列(8文字) → 文字ベクトル + 位置ベクトル → 次の文字を予測 → 答え合わせ → 重み調整
これを300回繰り返して賢くなる → 学んだ知識で文字を生成する
```

---

## 1. モデルの部品（43-45行目）

```python
tok_emb = nn.Embedding(vocab, n_embd)       # 9種の文字 → 8次元のベクトルに変換
pos_emb = nn.Embedding(block_size, n_embd)  # 8つの位置 → 8次元のベクトルに変換【新規】
net = nn.Linear(n_embd, vocab)              # 8次元のベクトル → 9種の文字のスコアに変換
```

コンピュータは文字を直接理解できないので、数値の列（ベクトル）に変換する必要がある。

- **tok_emb（Token Embedding）**: 「も」→ `[0.3, -0.1, 0.7, ...]` のような8個の数値に変換する辞書。最初はランダムな値で、学習で調整される。8個の数値に個別の意味はなく、8個セットで1つの文字を表す。学習が進むと、似た役割の文字は似たベクトルに、違う役割の文字は離れたベクトルになっていく
- **pos_emb（Positional Embedding）**: 「何番目の位置か」→ 8次元のベクトルに変換する。詳しくは後述
- **net（Linear）**: Embeddingの8個の数値を受け取り、「次の文字が う/か/た/は/ま/も/ら/れ/ろ である確率」を9個の数値（スコア）として出す

```
文字 → tok_emb → 8次元ベクトル ─┐
                                 ├→ 足し算 → net → 9個のスコア → 一番高いのが予測
位置 → pos_emb → 8次元ベクトル ─┘
```

### Linearの中身

Linearがやっていることは `出力 = 入力 × 重み + バイアス` という計算だけ。Embeddingの各要素に文字ごとの重みを掛けて足し合わせる:

```
「も」のEmbedding: [0.3, -0.1, 0.7, ..., -0.3]  ← 8個

「う」のスコア = 0.3×重み1う + (-0.1)×重み2う + 0.7×重み3う + ... + バイアスう
「か」のスコア = 0.3×重み1か + (-0.1)×重み2か + 0.7×重み3か + ... + バイアスか
「た」のスコア = 0.3×重み1た + (-0.1)×重み2た + 0.7×重み3た + ... + バイアスた
...9文字分繰り返す
```

この重み（8×9=72個）とバイアス（9個）= 計81個がLinearのパラメータ。`nn.Linear(8, 9)` と書いた時点でPyTorchがランダムな値で自動的に作ってくれる。最初はデタラメな予測しかできないが、学習を繰り返すことで「"も"のEmbeddingが来たら"た"のスコアを高くする」ような重みに育っていく

## 2. 損失関数とオプティマイザ（28-29行目）

```python
loss_fn = nn.CrossEntropyLoss()
opt = torch.optim.SGD(..., lr=1.0)
```

- **loss_fn（損失関数）**: 予測と正解のズレを1つの数値で表す。予測が正解に近ければ小さく、外れていれば大きくなる
  - **CrossEntropyLoss** は「複数の選択肢から1つを選ぶ」問題に使う損失関数。今回は9種の文字から正解の1文字を当てる問題なので、これが適している

### CrossEntropyLossの中身

正解率やスコア比較をしているわけではなく、**「正解の文字にどれだけ高い確率を割り当てられたか」** を見ている。

各文字にはアルファベット順に番号（インデックス）が振られている: う=0, か=1, た=2, は=3, ...

例えば「も」の次の正解が「た」（=インデックス2）だとして:

**① softmaxでlogitsを確率に変換**

```
logits:          [0.5, 0.2, 2.1, -0.3, 0.1, 0.8, -0.5, 0.3, 0.1]
                  う    か    た    は    ま    も    ら    れ    ろ

softmax後の確率: [0.05, 0.04, 0.52, 0.02, 0.03, 0.14, 0.02, 0.04, 0.03]
```

**② 正解の文字の確率だけ取り出す**

正解は「た」= インデックス2 なので、確率の2番目を取り出す → `0.52`

**③ -log を取る → これがloss**

```
loss = -log(0.52) = 0.65
```

正解の確率が高いほどlossは0に近づき、低いほど大きくなる:

```
正解に確率 1.0  → loss = -log(1.0)  = 0     （完璧）
正解に確率 0.5  → loss = -log(0.5)  = 0.69  （まあまあ）
正解に確率 0.01 → loss = -log(0.01) = 4.6   （全然ダメ）
```

学習はこのlossを下げようとするので、結果的に正解の文字に高い確率を出すようになる。実際のコードでは全入力文字のlossを平均して1つの数値にしている。
- **opt（オプティマイザ）**: 「どの方向にどれだけ重みを動かせばlossが下がるか」を計算して、実際に重みを更新する仕組み。`lr=1.0`は1回の更新の大きさ
  - **SGD（Stochastic Gradient Descent / 確率的勾配降下法）** は最もシンプルなオプティマイザ。「坂道で一番急な下り方向に一歩進む」のを繰り返して、lossが最も低い谷底を目指すイメージ

## 3. 学習ループ（56-65行目）

```python
for step in range(300):
    t = tok_emb(X)            # ① 文字→ベクトル
    p = pos_emb(positions)    # ② 位置→ベクトル【新規】
    x = t + p                 # ③ 合体【新規】
    pred = net(x)             # ④ 予測
    loss = ...                # ⑤ 正解との誤差
    opt.zero_grad()           # ⑥ 前回の勾配をリセット
    loss.backward()           # ⑦ 誤差逆伝播
    opt.step()                # ⑧ 重み更新
```

これを300回繰り返す。1回のループで起きること:

1. **予測**: 全入力文字をベクトルにして、位置情報を加えて、次の文字のスコアを出す
2. **誤差計算**: 予測スコアと正解を比べて「どれだけ間違っているか」を数値化
3. **勾配リセット（`zero_grad()`）**: 前回の勾配をゼロにする。PyTorchは`backward()`のたびに勾配を**足し込む**仕様なので、リセットしないと前回の勾配が混ざって正しく更新できなくなる
4. **逆伝播（`backward()` / バックプロパゲーション）**: 出力側から入力側に向かって「各パラメータ（=モデル内部の調整可能な数値）をどっちにどれだけ動かせばlossが下がるか」を計算する。この「動かすべき方向と量」が **勾配**
5. **更新（`step()`）**: 勾配の方向にパラメータを少し動かす

### backward() の中身

lossから逆順にたどって、各パラメータの勾配を求める。今回のモデルの計算の流れは:

```
Token Embedding + Pos Embedding → Linear → softmax → -log → loss
```

これを逆にたどる:

```
loss = -log(0.52) = 0.65
            ↑
    この0.52はsoftmaxの出力
            ↑
    softmaxの入力はlogitsの「た」= 2.1
            ↑
    この2.1はLinearの計算結果 = 0.3×重み1た + (-0.1)×重み2た + ...
            ↑
    じゃあ重み1たを少し増やしたらlossはどう変わる？ → これが勾配
```

例えば:

```
重み1たの勾配 = -0.2  →「増やすとlossが下がる」→ step()で増やす
重み1うの勾配 = +0.1  →「増やすとlossが上がる」→ step()で減らす
```

### 勾配はどうやって求めるのか

「+に動かしてみる → -に動かしてみる → 良い方を選ぶ」のではなく、**数式の微分から一発で求まる**。実際に値を動かして試す必要はない。

#### 微分のおさらい

微分は「ちょっと動かしたらどれだけ変わるか」の比率:

```
y = 3 × x  → dy/dx = 3     （xを1増やすとyは3増える）
y = x²     → dy/dx = 2x    （x=5なら、xを1増やすとyは約10増える）
```

#### 連鎖律（チェインルール）

計算が2段階あるとき、**1段階ずつの微分を掛け合わせればいい**。これが連鎖律:

```
① score = 0.3 × weight      （Linearの計算）
② loss  = score²             （損失の計算、簡略化した例）
```

「weightを動かしたらlossはどう変わるか」を求めたい:

```
d(loss)/d(weight) = d(loss)/d(score) × d(score)/d(weight)
```

それぞれは簡単に求まる:

```
d(score)/d(weight) = 0.3          ← score = 0.3 × weight を微分
d(loss)/d(score)   = 2 × score    ← loss = score² を微分
```

掛け合わせる:

```
d(loss)/d(weight) = 2 × score × 0.3
```

仮に `weight = 2.0` なら `score = 0.6` なので:

```
d(loss)/d(weight) = 2 × 0.6 × 0.3 = 0.36
```

勾配は `+0.36`（プラス）→ weightを増やすとlossが増える → step()で減らす。

#### このモデルでの実際の連鎖

段階が増えても同じで、各段階の微分を掛け合わせるだけ:

```
Embedding → Linear → softmax → -log → loss
    ④         ③         ②        ①

d(loss)/d(weight) = ①の微分 × ②の微分 × ③の微分
```

PyTorchは各段階の微分の公式を知っているので、これを順番に掛けていくだけで全パラメータの勾配が一度に求まる。

`step()` は `パラメータ = パラメータ - 学習率 × 勾配` を全パラメータに対して行う:

```
勾配 = -0.2 の場合: 0.4 - 1.0 × (-0.2) = 0.6  ← 増える
勾配 = +0.1 の場合: 0.4 - 1.0 × (+0.1) = 0.3  ← 減る
```

マイナスを掛けているので、勾配の符号と逆方向に動く = lossが下がる方向に自動的に動く。

PyTorchは `emb(X)` や `net(...)` を計算した時点で「どの計算をどの順番でやったか」を裏で記録しており、`backward()` はその記録を逆にたどって全パラメータの勾配を一気に計算する

繰り返すと loss が `2.5 → 0.4` と下がっていく = 予測が上手くなっていく。

## 4. 生成（70-84行目）

```python
idx = [to_i["も"]]  # 最初の1文字から始める
for _ in range(15):
    context = idx[-block_size:]                    # 直近8文字を取り出す
    x = torch.tensor([context])                    # (1, 文字数) のテンソルに
    t = tok_emb(x)                                 # 文字→ベクトル
    p = pos_emb(torch.arange(len(context)))        # 位置→ベクトル
    h = t + p                                      # 合体
    logits = net(h)                                # 次の文字のスコアを予測
    logits = logits[0, -1, :]                      # 最後の位置の予測だけ使う
    probs = F.softmax(logits, dim=-1)              # スコアを確率に変換
    next_i = torch.multinomial(probs, 1).item()    # 確率に従って1文字選ぶ
    idx.append(next_i)                             # 生成した文字を末尾に追加
```

- **logits**: モデルが出す生のスコア。まだ確率ではなく、大きいほど「その文字っぽい」という意味
- **softmax**: logits `[2.0, 0.5, -1.0, ...]` を確率 `[0.7, 0.15, 0.03, ...]` に変換する（全部足すと1.0になる）
- **multinomial**: その確率に従ってサイコロを振るように1文字選ぶ。確率0.7の文字は70%の確率で選ばれる

最小verとの違い:
- **複数文字を一度に入力**: 最大8文字の文脈を使って次の文字を予測する（最小verは直前の1文字だけ）
- **`idx[-block_size:]`**: 文字が8文字を超えたら直近8文字だけ使う（Positional Embeddingが8位置分しかないため）
- **`logits[0, -1, :]`**: 各位置で予測が出るが、生成では最後の位置の予測（= 次の新しい文字）だけ使う
- **`idx.append`**: 生成した文字を末尾に追加して、次のステップでは文脈が1文字増える（自己回帰生成）

## 5. Positional Embedding（位置の意味を学ぶ）

今回の最大の追加要素。なぜ必要で、何をしているのかを詳しく説明する。

### なぜ位置情報が必要か

Token Embeddingだけだと、「ももたろうはもも」の1番目の「も」と6番目の「も」がまったく同じベクトルになる。しかし:

```
"ももたろうはもも"
 ↑ 1番目の「も」→ 次は「も」
           ↑ 6番目の「も」→ 次は「も」（たまたま同じ）

"もたろうはももか"
 ↑ 1番目の「も」→ 次は「た」
          ↑ 6番目の「も」→ 次は「か」
```

同じ「も」でも、文中のどこにいるかで次に来る文字が違う。位置情報がなければ、モデルは語順を区別できない。

### 仕組み

```python
pos_emb = nn.Embedding(block_size, n_embd)  # 8つの位置 → 8次元のベクトル
```

位置0〜7にそれぞれ8次元のベクトルを割り当てる。Token Embeddingと同じサイズなので、足し算で合体できる:

```
1番目の「も」: tok[も] + pos[0] = [0.3, -0.1, ...] + [0.1,  0.2, ...] = [0.4, 0.1, ...]
6番目の「も」: tok[も] + pos[5] = [0.3, -0.1, ...] + [-0.2, 0.5, ...] = [0.1, 0.4, ...]
```

同じ「も」でも位置が違えば異なるベクトルになる。Positional Embeddingもランダムな値から始まり、学習で「意味のある位置表現」に育っていく。

### 学習データの形も変わった

最小ver: `(1文字, 次の1文字)` のペア → 13ペア

```
("も", "も"), ("も", "た"), ("た", "ろ"), ...
```

文脈付与ver: `(8文字, それぞれの次の1文字)` のペア → 6サンプル

```
入力: "ももたろうはもも"  →  正解: "もたろうはももか"
入力: "もたろうはももか"  →  正解: "たろうはももから"
入力: "たろうはももから"  →  正解: "ろうはももからう"
...
```

8文字の窓をスライドさせて学習データを作る。1サンプルに8位置分の予測が含まれるので、lossは全位置をまとめて計算する:

```python
loss = loss_fn(pred.view(B*T, V), Y.view(B*T))  # B×T個の予測をまとめて誤差計算
```

`view(B*T, V)` は形を `(サンプル数, 8, 9)` → `(サンプル数×8, 9)` に変形しているだけ。CrossEntropyLossが `(データ数, クラス数)` の形を要求するため。

## 6. マジックナンバーの意味

コード中に出てくる数値の意味:

### `block_size = 8` （13行目）

一度に見る文字数。「この文字の前8文字を文脈として使う」という意味。

- **大きくすると**: 長い文脈を使えるので予測精度が上がる可能性がある。ただし計算量とメモリが増える
- **小さくすると**: 短い文脈しか見えないので、遠くの文字との関係を学べない
- 今回は14文字のデータなので8で十分。実用的なLLMでは数千〜数万

### `n_embd = 8` （14行目）

各文字を表すベクトルの次元数。「も」を8個の数値 `[0.3, -0.1, ...]` で表現する。

- **大きくすると**: 文字の特徴をより細かく表現できるが、パラメータが増える
- **小さくすると**: 表現力が落ちるが軽い
- 今回は9種類の文字しかないので8で十分。実用的なLLMでは数千〜数万

### `1.0` （48行目: `lr=1.0`）

学習率（learning rate）。1回の更新で重みをどれだけ動かすか。

- **大きすぎると**: 最適な値を飛び越えてしまい、lossが発散する
- **小さすぎると**: 学習が遅くなり、なかなか賢くならない
- `1.0`はSGDにしては大きめだが、今回はデータが小さいのでこれで問題ない

### `300` （56行目: `range(300)`）

学習の繰り返し回数（エポック数）。

- **多すぎると**: 学習データを丸暗記してしまう（過学習 = 学習データでは完璧だが未知のデータに対応できなくなること）。あと時間がかかる
- **少なすぎると**: まだ学習が足りず、予測がランダムに近い
- 今回はlossが200ステップ辺りで十分下がっているので300で十分

### `15` （73行目: `range(15)`）

生成する文字数。単純に「何文字出力するか」の指定。好きな数に変えてOK。
